Cilium VXLAN 流量分析

Cilium 提供多种组网模式，默认情况下使用 VXLAN 模式，通过 Overlay 组网的方式实现 Pod 的跨节点通信。

Cilium 在基于 VXLAN 的 Overlay 组网情况下，默认使用 UDP/8472 进行通信，在宿主机新增了四个虚拟网络接口：

  • cilium_vxlan：对数据包进行 VXLAN 封装和解封装操作

  • cilium_net 和 cilium_host：是一对 veth-pair，cilium_host 作为该节点所管理的 ClusterIP 子网的网关

  • lxc_health：用于节点之间的健康检测

需要注意的是，虚拟化平台有可能会占用同样的端口（如深信服）用于 VXLAN 通信，当端口冲突的时候，虚拟化平台为了防止 VXLAN 报文出现环路（即从业务口出去，从 VXLAN 环进来），而不转发 8472 端口的报文，而是直接被 DROP 掉，从而导致 Pod 跨节点通信失败。

有时候可能使用连通性探测工具（如 nc）探测是能通的，原因是连通性探测的报文较小，长度小于 VXLAN 业务报文长度，不会被识别成 VXLAN 业务报文。

同节点 Pod 通信

当 Pod 使用容器网段启动的时候，Cilium 会为其分配地址并配置路由，默认路由会指向宿主机的 cilium_host 网卡，而默认情况下 ARP 表为空：

当同节点 Pod 之间进行通信的时候，网络走向如下：

busybox-s2q6k 的 IP 为 10.42.0.97，nginx-596795d4c-zw7x6 的 IP 为 10.42.0.182，当 busybox-s2q6k 发送 ICMP 请求到 nginx-596795d4c-zw7x6 的时候，经过路由表，查询到下一跳为 10.42.0.26，即宿主机的 cilium_host 网卡，此时 Pod 会发送 ARP 广播，获取该 IP 的 MAC 地址，但 cilium_host 网卡是 NOARP 状态的，也就是不会响应 ARP 报文。

Cilium 会根据 Pod 对应的 veth-pair 设备，将其 MAC 地址返回给 Pod，也就是 busybox-s2q6k 最终会获取到 lxcee24765fd490 的 MAC 地址，并进行缓存，随后再发送 ICMP 报文：

会发现所有容器网段的路由都是指向 cilium_host 网卡，而 cilium_host 会使用 eBPF 实现转发逻辑。

busybox-s2q6k 的 IP 为 10.42.0.97，busybox-9pzgx 的 IP 为 10.42.1.64，同理，Pod 的流量会先到达对应的 veth-pair 设备，根据目的地址查询宿主机路由表，而目的地址也属于容器网段 IP，所以会由 cilium_host 进行处理。

10.42.1.64 匹配第二条规则，随后将 ICMP 报文发送给 cilium_vxlan 处理。cilium_vxlan 网卡会进行 VXLAN 封装，封装后再通过 Underlay 进行路由，也就是将报文转发给宿主机物理网卡，再发送出去。

此时的报文源地址和目的地址不在是容器网段的地址，而是两台节点的物理地址。
